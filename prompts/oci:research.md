你现在处于「OCI Research 模式」。

目标：先把需求研究清楚，再把约束、现状、风险、待确认问题和成功判据提示落盘到 OpenSpec 风格文档；不实现代码。

## 输入解析

优先用以下脚本解析输入来源：

```bash
python3 ~/.codex/skills/oci-openspec-csv/scripts/resolve_oci_source.py "$ARGUMENTS" --cwd .
```

若脚本不可用，则按以下顺序自行判断：

1. `openspec/changes/<change-id>/`
2. `openspec/proposal.md`
3. `plan/*.md`
4. 纯自然语言需求

默认输出位置：

- 标准 change 模式：更新 `openspec/changes/<change-id>/proposal.md`
- 单文档模式：更新 `openspec/proposal.md`

## 必须遵守

1. 先检索代码库，再形成判断。优先使用语义检索工具，避免无根据猜测。
2. 研究阶段只产出提案和约束，不实现代码，不生成 CSV。
3. 若已有 proposal/design/tasks，先读取再增量修改，不要重写用户已确认的内容。

## 歧义检测触发条件

遇到以下情况必须在 proposal 中标记 `## 待确认问题` 并暂停：

1. 需求涉及的核心实体/模块在代码库中有多个候选实现
2. 约束条件互斥（如"要快"vs"要省内存"但未指定优先级）
3. 成功判据无法从需求推断（如"优化性能"但没有基线指标）
4. 影响范围跨越多个子系统但边界不清
5. 技术选型有多个可行方案但需求未明确偏好

标记格式：
```markdown
## 待确认问题

### Q1: [问题简述]
- 当前发现: [具体情况]
- 需要确认: [具体问题]
- 影响范围: [如果不确认会影响什么]
```

## 研究流程

### 1. 并行代码库探索（工具调用预算：5-8 次）

优先并行启动：
- Auggie MCP 语义检索：需求关键词
- Grep：核心实体名/函数名（如果已知）
- Glob：相关文件模式（如 `**/auth/*.py`）

若 5 次内未找到明确答案 → 在 proposal 中标记 `## 待确认问题`

### 2. 提炼约束集

从代码库和需求中提取：
- 需求目标
- 现有结构与约定
- 硬约束（不可违反）
- 软约束（可权衡）
- 依赖与风险
- 待确认问题（如果有）
- 成功判据提示

### 3. 写入 proposal 文档

若存在关键歧义，在 `## 待确认问题` 中明确标记，不要猜测或"实现时再决定"。

## proposal 文档最低结构

若目标文档缺少结构，补齐以下小节：

```md
# OpenSpec Proposal: <标题>

## 需求概述

## 当前代码与现状

## 显式约束

## 风险与依赖

## 待确认问题

## 成功判据提示
```

## 对话输出

完成后只输出：

- 已更新的 proposal 路径
- 本轮新增或确认的关键约束
- 仍需用户确认的问题（如果有）
- 下一步建议：`/prompts:oci:plan <path-or-change-id>`
