你现在处于「OCI Plan 模式」。

目标：把 proposal 收敛成后续可机械执行的计划。任何重要决策未冻结前，不进入 CSV 执行阶段。

## 输入解析

优先运行：

```bash
python3 ~/.codex/skills/oci-openspec-csv/scripts/resolve_oci_source.py "$ARGUMENTS" --cwd .
```

支持以下来源：

- `openspec/changes/<change-id>/`
- `openspec/proposal.md`
- 任意明确的 `proposal.md`

## 必须遵守

1. 先读已有 proposal，再补设计和任务，不要凭空重写。
2. 任何影响实现路线的歧义都必须标记到 `## 待确认问题`，不允许”实现时再决定”。
3. 计划阶段只写文档，不实现代码。
4. 若是标准 change 模式，优先更新：
   - `proposal.md`
   - `design.md`
   - `tasks.md`
   - 必要时更新 `specs/*/spec.md`
5. 若是单文档 proposal 模式，则在同一文档中补齐零决策任务流、PBT、成功判据。

## 零决策检查清单

在输出 plan 前，必须确认以下问题都有明确答案：

- [ ] 每个任务的输入文件路径是否具体到 `path:line` 或明确的模块名？
- [ ] 每个任务的验收标准是否可以用命令或断言验证？
- [ ] 是否存在”根据实际情况决定”的表述？（如有则未冻结，需标记到 `## 待确认问题`）
- [ ] 任务间依赖是否显式声明？
- [ ] 是否有”可能需要”的模糊范围？（如有则需收敛或标记）

若检查清单有未通过项，在文档中补充 `## 待确认问题` 并暂停。

## 计划阶段最低产物

至少要冻结以下内容：

- 明确的范围边界和非目标
- 关键技术决策与参数
- 文件/模块级影响范围
- 可执行的阶段化任务流
- 验收标准
- 关键回归点
- 适用时的 PBT 或系统不变量

## 推荐结构

如果是单文档 proposal，补齐这些章节：

```md
## 零决策任务流

## PBT（Property-Based Testing）属性

## 成功判据

## 风险与缓解
```

## 交互规则

当出现以下情况时必须先问用户：

- 两种以上实现路线都可行，但影响不同
- 成功判据无法从现有文档推断
- 文档中的路径、模块名、接口名不够具体
- 计划会扩展到用户没有明确要求的新范围

## 收尾要求

计划完成后输出：

- 已更新的文档路径
- 本轮冻结的关键决策
- 仍未解决的问题（如果有）
- 是否已经适合生成 CSV
- 下一步建议：`/prompts:oci:openspec_to_csv <path-or-change-id>`
